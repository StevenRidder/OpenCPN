## Sanitization Checklist

Before PUB-2/PUB-3 publishable branches or docs are cut:

- Secrets: no tokens, API keys, credentials, certificates, `.env`, `.pem`, or
  `.key` files.
- Private chart data: no private ENC cells, S-63 permits, oeSENC data, CM93
  bundles, commercial basemaps, generated SENC caches, or private voyage data.
- Generated artifacts: no CMake build trees, generated PNG baselines, generated
  renderer cache output, local demo logs, or runtime cache directories.
- Sample data: any committed PMTiles/MBTiles/demo data must be license-safe,
  documented as sample data, and not required to understand the proof branch.
- Local paths: docs may use `/tmp/...` as disposable examples, but public setup
  must not require `/Users/...`, `/private/...`, Dropbox paths, or Steve's
  machine layout.
- Live runtime: no public command may start, stop, or depend on the live Helm
  `:8080` screen. Use private ports for demos and verification.
- Claims: state that this is a POC, not full S-52 parity, not ECDIS
  certification, not a primary-navigation recommendation, and not a full WebGPU
  or Metal renderer implementation.
- Framing: ask for architecture review. Do not call the current OpenCPN renderer
  obsolete, do not frame this as a wholesale replacement, and do not imply
  upstream should accept a large web-client direction.
- Helm framing: Helm shares the nautical pipeline, neutral model, inspection
  metadata, and scheduler policy. Helm product UX is WebGPU-first with
  WebGL/MapLibre and server-raster fallbacks. Helm does not consume VSG as its
  client renderer.
- OpenCPN framing: VSG/Vulkan is the native proof backend only. The legacy
  renderer remains default unless a feature flag routes a validated neutral
  model into the shared renderer.
- Metal framing: Metal is deferred compatibility only. Do not present Metal as a
  current implementation target or product priority.
- Module boundaries: keep chart-source normalization, presentation compiler,
  neutral render model, scheduler policy, backend cache, adapters, and support
  scripts visibly separate. Avoid any monolithic rewrite impression.
- Language/runtime: renderer/runtime implementation stays C++17-shaped and
  OpenCPN-native. Shell scripts and JSON fixtures are support artifacts only.
- Agent clutter: do not lead public docs with agent names, board status, or
  autogenerated-looking branch churn. Preserve PR evidence, but write the RFC as
  a human-readable architecture package.
- License and provenance: keep OpenCPN-derived code GPL-compatible and
  traceable; keep Helm web/backend licensing and OpenCPN GPL boundaries explicit.

## Findings

- `chart-render/` on `vulkan/render-core-poc` contains the intended C++/CMake
  proof surfaces, JSON fixtures, docs, and support script. The audit did not find
  committed secrets, private chart cells, S-63/oeSENC artifacts, or generated
  CMake/build cache output in the proof subtree.
- The proof docs intentionally contain `/tmp/...` example paths for disposable
  build directories and generated demo summaries. This is acceptable for
  developer commands, but PUB-2/PUB-4 should keep them as examples rather than
  presenting them as required machine paths.
- `chart-render/docs/STAKEHOLDER_DEMO.md` was harmonized in PUB-2: WebGPU is
  Helm's client direction, Metal is deferred compatibility only, and neither is
  claimed as implemented by the current OpenCPN proof branch.
- `chart-render/docs/HELM_WEB_RENDER_TARGET.md` contains the corrected
  WebGPU-first Helm framing and should be the source for PUB-3.
- Helm `main` has broad planning and business docs that mention AI, cloud,
  native Metal, live ports, and older audit findings. Do not treat the whole
  Helm repo as the Vulkan proof package. PUB-3 should curate a narrow
  Helm/WebGPU explanation.
- Helm `main` includes committed demo/sample map assets, including a PMTiles
  sample. PUB-3 must verify the license/source posture before linking or
  packaging any sample asset with the Vulkan proof.
- Helm README already says private charts, private basemaps, credentials,
  generated caches, and vessel-sensitive data must stay out of Git, and warns
  shared developers not to use live `:8080`. Preserve that stance.
- OpenCPN PR #22 is still open and dirty at audit time. It is not a blocker for
  PUB-1, but a final public branch should not leave the Metal compatibility note
  half-landed or stale.

## Required Follow-Up

PUB-2 must prepare the sanitized OpenCPN proof branch:

- Rebase or wait for ADAPT-5 PR #22, or explicitly exclude it.
- Keep `chart-render/` focused on reviewable C++ core, docs, fixtures, and
  support scripts.
- Ensure README/acceptance/demo docs use WebGPU-first Helm language and
  deferred-Metal language consistently.
- Keep generated evidence out of Git; regenerate it from commands.

PUB-3 must prepare the sanitized Helm/WebGPU explanation:

- Do not publish the entire Helm planning corpus as the Vulkan proof.
- Use Helm main only to show REPO-4/REPO-5 evidence and the consumer/product
  target framing.
- Explain WebGPU-first client rendering, WebGL/MapLibre fallback, server-raster
  fallback, offline/cache/prefetch, and object inspection.
- Verify any sample PMTiles/MBTiles or screenshots before including them.

PUB-4/PUB-5 must package and communicate the proof:

- Build the RFC from curated OpenCPN proof docs plus curated Helm/WebGPU docs.
- Ask maintainers for architecture review and seam feedback.
- List limitations honestly: no ECDIS certification, no full S-52 parity, no
  production WebGPU renderer, no production Metal backend, and no standalone
  renderer repository decision.
